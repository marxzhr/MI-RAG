# 三、你现在应该采用的最终 pipeline

## 整体流程

### Step 1. 用户输入原始问题

例如：  
“糖尿病人能不能吃阿司匹林，会不会有副作用？”

---

### Step 2. 关键实体提取

从 query 中提取：

* 疾病：糖尿病
* 药物：阿司匹林
* 需求点：能不能吃 / 副作用

这一步是整个系统的起点。

---

### Step 3. 基于关键实体做多路重写

不是只重写一条，而是重写 3~5 条。

例如围绕上面这个问题，可以生成：

1. 糖尿病患者是否可以服用阿司匹林
2. 阿司匹林对糖尿病患者的适用性
3. 糖尿病人服用阿司匹林的副作用与禁忌
4. 阿司匹林在糖尿病患者中的使用注意事项

老师强调的就是这个：  
**一条 query 变成多条 query**

因为多路 rewrite 会让检索覆盖更全。

---

### Step 4. 多查询并行检索

把这 3~5 条 query 全部 embedding，然后一起去向量库里捞。

这一步的优势是：

* 原始 query 可能表达模糊
* 单条 query 可能漏召回
* 多路 query 可以覆盖不同表达方式
* 提高召回率

---

### Step 5. 合并召回结果

把多条 query 检索到的候选文档合并。

这里要做：

* 去重
* 合并相似 chunk
* 保留更丰富候选集

---

### Step 6. rerank 重排

用 CrossEncoder 或 reranker 对候选文档进行相关性排序。

---

### Step 7. 构造 prompt，送入 LLM 生成

将：

* 原始问题
* 提取出的关键实体
* 最终排序后的检索结果

一起送给大模型生成答案。
